Qui controle le client ?

La bataille des protocoles est une bataille pour le controle de la relation client.

  • Protocole ouvert (UCP) : le marchand garde la relation directe
  • Plateforme fermee (ACP/Amazon) : un intermediaire s'installe

Le choix de protocole est un choix strategique, pas technique.

Chaque etape = un endpoint REST ou un outil MCP. L'agent progresse de maniere autonome et structuree.

Le flux UCP pas a pas

1. GET /.well-known/ucp → Decouverte des services, capabilities, payment handlers

2. GET /api/products → Catalogue : produits, prix, disponibilite

3. POST /checkout-sessions → Creation du panier (line_items)

4. PUT /checkout-sessions/{id} → Ajout buyer, discounts, fulfillment

5. PUT /checkout-sessions/{id} → Selection adresse + option livraison

6. POST /checkout-sessions/{id}/complete → Paiement → Commande creee

7. GET /orders/{id} → Suivi de la commande

Le consentement voyage avec la commande

Le checkout hash

Mecanisme anti-autonomie : l'humain garde le controle

1. Le serveur genere un checkout_hash a chaque mise a jour

2. Le hash encode : produits + prix + livraison + reductions

3. L'agent doit presenter ce hash a l'humain pour approbation

4. Le hash est requis pour finaliser la commande (via MCP)

L'agent peut tout faire SAUF payer sans le consentement explicite du buyer

DEMO: UCP Discovery

Le point d'entree de tout agent UCP :

GET /.well-known/ucp

  • Version du protocole
  • Services disponibles
  • Capabilities (checkout, order, discount...)
  • Payment handlers

Equivalent du robots.txt pour les agents IA

DEMO: Catalogue

L'agent explore le catalogue :

GET /api/products

  • 6 produits (fleurs)
  • Prix en centimes (USD)
  • gardenias : rupture de stock (qty=0)
  • Donnees structurees, pas du HTML

DEMO: Creer un Checkout

L'agent cree un panier :

POST /shopping-api/checkout-sessions

  • bouquet_roses x2 = $70.00
  • orchid_white x1 = $45.00
  • Status attendu : incomplete
  • Le serveur calcule les totaux

DEMO: Promo + Adresse

L'agent enrichit le checkout :

PUT /checkout-sessions/{id}

  • Buyer : john.doe@example.com
  • Promo : 10OFF (10% de remise)
  • Fulfillment : shipping
  • Adresses auto-populees (2 connues)
  • Livraison gratuite (bouquet_roses inclus)

DEMO: Selection Options

L'agent finalise la configuration :

PUT /checkout-sessions/{id}

  • Selectionne l'adresse de livraison
  • Selectionne l'option de livraison
  • Status attendu : ready_for_complete
  • IDs extraits dynamiquement de l'etape precedente

DEMO: Finaliser la commande

L'agent complete le checkout :

POST /checkout-sessions/{id}/complete

  • Payment handler : mock_payment_handler
  • Token : success_token
  • Resultat : commande creee
  • Order ID genere

DEMO: Dashboard Temps Reel

Le dashboard du marchand, mis a jour en temps reel via SSE (Server-Sent Events)

Impact economique

Reduction SAV : 60-80%

  • Les cas simples (suivi, retours, echanges) sont entierement automatises
  • Les cas complexes sont pre-qualifies par l'agent avant escalade humaine
  • Le cout marginal d'une resolution tombe a quasi zero

Le SAV n'est plus un centre de cout. C'est un levier de fidelisation automatise.
